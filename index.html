How a request flows through the system

A live trace of POST /analyze — résumé + job description in, evidence-based gap report out. Watch where each step's reasoning is routed: sensitive data stays local; only de-identified, public data is ever eligible for a cloud model.

Authored by Raghava Kundavajjala
What it is — a local-first engine that turns a résumé + a job description into an evidence-grounded skill-gap report: it scores every required skill against your real career history, refuses to claim anything it can't cite (no evidence → the skill is forced to 0), rewrites résumé bullets to surface transferable experience, and parses your résumé on-device so it never leaves your machine. A separate Advocate/Skeptic/Judge debate independently re-checks the critical gaps — running entirely on-device. Below is a live trace of how one analysis request flows through all six layers.

Speed
🔒 Local-first (ADR-007)
Résumé & career data are routed to on-device Ollama only — never to any cloud API, in any mode.
🚫 No hallucinated skills
Pydantic-validated output; a skill with no cited evidence is forced to score 0.
⚖️ Agentic debate verify
Critical gaps are re-checked by an Advocate/Skeptic/Judge debate; the model decides how many rounds to run (≤3), entirely on-device.
💰 Budget hard-stop
Spend is checked before every paid call; on the cap, calls fall back to local Ollama — analysis never fails.
💾 Crash-safe writes
Every persist is write-temp → fsync → atomic rename; a crash never corrupts a file.
🧵 Thread-safe by request
LLM mode is read from per-request state, never global config — concurrent calls don't interfere.
Strict one-way layering: frontend → api → services → repositories / agents → domain · LangGraph pipelines · sensitivity-aware LLM router (Ollama · Claude · OpenAI) · FastAPI · Python 3.14
